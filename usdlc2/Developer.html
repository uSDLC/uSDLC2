Source Editor Selection Panel

The uSDLC tab, context menu and Alt-V hot-key will all bring up a project source editor selection panel (unless the cursor is on a section link reference). From this panel you can choose a file by:

  • Selecting from the file system tree
  • Typing in part of the name
  • Search for file contents (grep)

Code Editing

Key Mapping

CodeMirror provides key maps for Emacs, Vi and CodeMirror modes - where the latter is similar to TextMate. Insert mode while using the Vi map is identical to the TextMate mode, providing the best of both approaches.

Remembering key bindings is always difficult. To aid in this the uSDLC context menu displays the key mapping against the operation - when you hover over the entry of interest.

Ref: codemirror.coffee

Console

For project interaction where uSDLC does not provide a function, use the built-in terminal accessible from the uSDLC tab. This console links to your system shell, but not all commands are available. Because the link is not a real terminal, programs like Vi and ssh do not work.

Ref: terminal.coffeeshell.coffee

Pairing

uSDLC supports pairing (2 or more) in the browser. There is one master copy. Once others have joined by name, any can interact with the programs. All users must have access to the same uSDLC2 server - either public host, local network or VPN.

Ref: ckeditor.coffeeterminal.coffeepairing.coffee

Setup

Any uSDLC2 browser instance can become a pair master. Choose the pair icon under the uSDLC2 tab, select master and give the session a name. Others can join the session by following the same procedure with master deselected. Clients will need to know the name of the master they wish to join. Any additional security can be provided by network and vpn security measures.

Initialisation

When a client pair session is established, the current page is replaced by the windows opened on the master. Browsers do not need to be the same size.

Synchronisation

Unlike an independent session, no changes are made to disk by clients - all are sent to the master for processing. Commands are published and consumed when:

  • editor changes occur
  • text is highlighted
  • Change of project or document
  • uSDLC tab requests such as code, bridge, file viewer, etc.
  • windows are opened
  • contents of the console or play windows change

All participating browsers will see the same information, but it may be in different formats based on browser choice and user action.

Interaction

Selecting windows and scrolling happen locally and do not reflect on the other browsers. All paired sessions will move to the same place in the same window if:

  • A new window is opened
  • Text is highlighted
  • A changed is made to an editor
  • A play command is triggered
  • typing in the terminal window

With this approach, review is possible with focus returning to the task at hand when changes are indicated.
