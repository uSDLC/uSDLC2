Preface

Before uSDLC can help us with the design we need to have a high level concept on how our project will look. Without this our design process lacks direction. There is a process of consideration of different options and decisions that need to be made first. If we document the process here they provide a starting point for the design and a perspective on the decisions made for a clearer understanding of the project.

Reason - Result - User Interface - Language - Frameworks - Deployment

Reason - notes of why this project is needed

Through my career I have worked in most positions within a software development project. As a software manager I was particularly incensed by the lack of connectivity between the different processes that make up a life-cycle. Examples presented to the analysts does not get recorded for development or testing. The 'why' of a process or approach is not communicated to teams downstream of the analysts and architects. Testers and document writers often have to independly learn how an application operates before they can do their jobs - often making inaccurate assumptions. Support rarely get enough information to reliably keep the system updated. Lastly once a project is done the information needed for the next phase is often lost or out of date - making a complete rewrite more likely than upgrading the current system.

Paul Marrington (paul@marrington.net)

Investigation - the path to a decision

I have spent the last two years creating a proof-of-concept. It is available on GitHub. During that time it has collected a lot of cruft as I researched avenues and technologies that proved not to be on the main path. I also decided on supporting frameworks that are either less suitable than I hoped or have been surpassed by others. The product is complete except for some really detailed documentation. I could not use uSDLC to develop uSDLC, but I can use it for uSDLC2.

Result - a high level description of what the finished project will look like

That's easy. Look at http://usdlc.net. uSDLC2 will have similar functionality but with a cleaner interface and running on a smaller more responsive framework.

User Interface - technology, look and feel, user experience

uSDLC uses a single page browser based UI. Content is loaded on demand to provide a responsive flexible interface.

Language - computer languages and libraries decided upon

uSDLC2 relies on coffee-script for both client and server core code. With the advent of V8, the Google Javascript engine, Javascript now has mainstream performance. I find coffee-script the cleanest of the modern interpreted languages.

Frameworks - hardware and software frameworks anticipated

On the server side, Node.js has progressed in leaps and bounds. With coffee-script to write in I now consider it an excellent platform. In uSDLC2 the server will be more light-weight - primarily serving files and acting as an instrumentation bridge to production code.

The client requirse a HTML-5 browser such as Chrome to leverage the modern open source packages and to provide client side persistence.

HTML wysisyg editing uses Aloha for a more integrated and customisable solution.

Source and data editing are be provided by Ace - the core source coded editor for Cloud9.

jQuery is, as alway, provide the DOM manipulation.

Advanced user interface sub-systems are provided by the GPL portions of DHTMLX.

Deployment - envisioned deployment environment

One other benefit of node.js is that it is a single executable or around 10Mb for each environment. The online installer can be a archive with the node.js binary for the target environment and a simple Javascript installation script. uSDLC, once running, has a utility to bundle up an existing installation for off-line clones.
